# DBT Project Structure for Global ID Integration

## Context

This ADR documents a proposal for how the harmonization pipeline handles
global IDs, regardless of how IDs are minted. The same pipeline architecture
should be used whether IDs are generated by the current minting script or by
another method in the future.

The current KF approach was introduced on 2026-05-21 during the
DFF 2.0 Combined Code Review and Working Demo Meeting.

A script (referred to here as the minting script) currently:

- connects to the KF warehouse
- pulls descriptor data from a study manifest view
- requests global IDs from the dewrangle API
- writes output to a warehouse table used for ID lookup

In this ADR:

- The `manifest` is the input data for ID generation (CSV or warehouse-backed)
- The `global identifiers` table is the script output table that stores descriptors
  and global IDs
- The `stb model` is an internal harmonization stage model and not the generic
  stable layer name used elsewhere

Pipeline stages:

- src (source)
- int (intermediate)
- stb (study-level normalized stage)
- combined
- access
- export

## Decision

Adopt a pipeline pattern where global ID assignment is treated as a distinct
step between descriptor preparation and completion of study-specific models,
while keeping existing stage boundaries and downstream architecture stable.

The pipeline must support any minting implementation as long as it can:

1. consume a study-level `manifest` input
2. return or persist global IDs in a defined lookup table
3. preserve deterministic joins back to study entities

## Advantages

- Preserves the current pipeline structure and existing stage semantics.
- Separates harmonization logic from global ID assignment.
- Allows per-study flexibility for descriptor preparation stage.
- Enables a common ID-generation workflow across studies.
- Supports reusable templating with dbt macros and pipeline-utils.
- Aligns dev and production minting workflow.
- Uses manifest views to expose required descriptor fields only.

## Additional Specifications

### Global ID Stabilization and Storage

- Prefer view-based manifest inputs when requirements allow.
- Evaluate incremental models or snapshots if persistence/versioning is needed.
- Final storage strategy should be revisited when minting-script requirements
  are fully specified.

### Manifest Model

- Combines all descriptor fields requiring global ID generation for one study.
- Candidate dbt materializations:
  - view
  - incremental
  - snapshot
- location: `{org}_{study_id}_int.global_id_manifest`

### Global Identifiers Table

- Stores minting output and should mirror manifest descriptors plus global IDs.
- Candidate dbt/source materializations:
  - table
  - incremental
  - snapshot
- Proposed location: `{org}_{study_id}_int.global_id`

## Consequences

- Positive:
  - Global ID integration becomes consistent across studies.
  - Architecture remains stable even if minting implementation changes.
